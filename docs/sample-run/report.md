## 1. Executive summary

The audited build (`demo-app/src/server.js`) is **not ship-ready**. It contains
**2 critical** and **5 high** severity issues, including a SQL-injection
authentication bypass, an unauthenticated admin endpoint that leaks password
hashes, multiple injection points, and reflected XSS. A hardened build already
exists at `demo-app/hardened/server.js`; an authorized localhost red-team
confirmed every issue is **blocked** there.

**Findings by severity (vulnerable build → hardened build):**

| Severity | Vulnerable | Hardened |
|----------|:----------:|:--------:|
| Critical | 2 | 0 |
| High     | 5 | 0 |
| Medium   | 3 | 0 |
| Low      | 0 | 0 |
| Info     | 0 | 0 |

**Top risks (plain language):**
1. Anyone can log in as any user without a password (SQL injection on `/login`).
2. Anyone can list all users and their password hashes with no login (`/admin/users`).
3. Anyone can read any user's private notes, and tamper with the notes query (IDOR + injection).

**Compliance snapshot:**

| Framework | Covered | Gaps |

**Residual risk at Standard tier:** Covers the OWASP Top 10 for this app
surface. Does **not** include supply-chain hardening, infra/IaC review, or

## 2. Accessibility map (what is accessible vs. not)

### Vulnerable build — publicly accessible (no auth)
| Surface | AuthN | AuthZ | Sensitivity | Intended? | Finding |
|---|---|---|---|---|---|
| `POST /login` | none | n/a | confidential | ⚠ no rate limit / SQLi | F-001, F-010 |
| `GET /search` | none | n/a | public | ⚠ reflects input | F-007 |
| `GET /notes/:id` | **none** | **none** | confidential | ❌ should be owner-only | F-003, F-004 |
| `GET /admin/users` | **none** | **none** | secret/PII | ❌ should be admin-only | F-002 |

### Hardened build — after remediation
| Surface | AuthN | AuthZ | Status |
|---|---|---|---|
| `POST /login` | none (issues token) | rate-limited 5/min | OK |
| `GET /search` | none | output-encoded | OK |
| `GET /notes/:id` | Bearer token | ownership-checked | OK |
| `GET /admin/users` | Bearer token | role=admin, no hash exposure | OK |

### Correctly NOT accessible (hardened)
- SQL-injection auth bypass on `/login` — neutralized by parameterization.
- Cross-account note access — blocked by ownership check (403).
- Admin user list as a normal/anon user — blocked (401/403); hashes never returned.
- Stack traces — no longer leaked (generic 500).

## 4. Methodology
- **Tier & frameworks**: Standard / OWASP Top 10 2021 + ASVS + API Top 10 2023.
- **Scope**: in-scope `demo-app/src/server.js` and localhost:3000; out-of-scope
  anything non-local.
- **Authorization basis**: self-attested owner; localhost-only demo.
- **Process**: manual code review against the Fortify attack catalog + an
  authorized live localhost red-team (this session) that exploited each issue on
  the vulnerable build and confirmed it blocked on the hardened build.
- **Phasing**: discovery → audit/threat-model → report → (hardening demonstrated
  in `hardened/`) → red-team verification → report (per NIST SP 800-115).

## 5. What changed this run
Report-first run — `src/server.js` was intentionally left vulnerable (it is the
demo target). All remediations are demonstrated in `demo-app/hardened/server.js`
(parameterized queries, argon2id, ownership/role checks, output encoding, helmet
headers, rate limiting, generic errors, secret via env).
